# Техническое описание проекта reco.kz

## Применяемые технологические решения

- **Full-stack веб-приложение** на базе Next.js (App Router) с единым репозиторием: фронтенд и бэкенд-логика в одном проекте.
- **Типобезопасный API** через tRPC: типы процедур и клиента генерируются из серверного кода, что снижает ошибки интеграции.
- **Серверная и клиентская валидация** входных данных с помощью Zod; единая схема для форм (react-hook-form + zodResolver) и для API.
- **ИИ для обработки документов**: определение типа документа, банка, структуры таблицы (строка заголовков, колонки) и помощь в сверке банковских и CRM-транзакций (Vercel AI SDK + OpenAI).
- **Шифрование чувствительных данных**: учётные данные Rekassa хранятся в БД в зашифрованном виде (модуль `server/lib/encryption`).
- **Мультитенантность по организациям**: привязка пользователей к организациям через Clerk (metadata), все данные изолированы по `organizationId`.
- **Реалтайм-команды** для создания платежей через Pusher (канал по организации, событие создания платежа).

## Этапы реализации продукта (пошаговая схема)

1. **Регистрация и онбординг**  
   Регистрация/вход через Clerk (в т.ч. SSO), выбор/создание организации, заполнение профиля (onboarding).

2. **Создание отчёта**  
   Пользователь создаёт отчёт за период; в БД создаётся запись `Report` со статусом `import_info`.

3. **Ввод базовой информации по отчёту**  
   Шаг «Импорт информации»: даты начала/окончания, кассовый остаток; статус отчёта — `import_info`.

4. **Импорт банковских документов**  
   Загрузка Excel (выписки, отчёты о продажах); парсинг через xlsx + ИИ (тип документа, банк, структура таблицы); создание `Document` (type: bank) и `Transaction`; статус — `import_bank`.

5. **Импорт CRM-документов**  
   Аналогичная загрузка и парсинг CRM-выгрузок; создание документов типа `crm` и транзакций; статус — `import_crm`.

6. **Продажи и расходы**  
   Шаги «Продажи» и «Расходы» (статусы `sales`, `expenses`): настройка типов операций и связей с банковскими/CRM-транзакциями.

7. **Сверка (реконсиляция)**  
   Сопоставление банковских и CRM-транзакций (по сумме, дате, идентификаторам); создание записей `Reconciliation`; при необходимости — использование ИИ для предложений сверки.

8. **Интеграция с Rekassa**  
   Сохранение зашифрованных учётных данных Rekassa в `Conf`; отправка платежей через API/команды (Pusher); приём вебхуков (при необходимости).

9. **Завершение отчёта**  
   Статус отчёта переводится в `done`; отчёт доступен в списке и в детальном просмотре (транзакции, сверки, типы операций).

## Архитектура проекта

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              КЛИЕНТ (Browser)                            │
│  Next.js App (React 19) · App Router · Tailwind · Radix UI · nuqs       │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    │
                    ┌───────────────┴───────────────┐
                    │  tRPC Client (React Query)    │
                    │  /api/trpc/[trpc]             │
                    └───────────────┬───────────────┘
                                    │
┌───────────────────────────────────▼─────────────────────────────────────┐
│                         СЕРВЕР (Next.js API / Route Handlers)             │
│  createTRPCContext (Clerk auth, organizationId) · protectedProcedure     │
├───────────────────────────────────────────────────────────────────────────┤
│  Роутеры: organization | reports | documents | reconciliation            │
│           | transaction | transactionType | bankTransaction | crmTransaction│
│           | rekassa | webhook | user                                       │
├───────────────────────────────────────────────────────────────────────────┤
│  Вспомогательные сервисы: ai (detectBank, detectTable*, reconcileDocs)    │
│  encryption · mergeKnpTransactions · parse · parseDateTime · parseAmount  │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    │
        ┌───────────────────────────┼───────────────────────────┐
        │                           │                           │
        ▼                           ▼                           ▼
┌───────────────┐         ┌─────────────────┐         ┌─────────────────┐
│  PostgreSQL   │         │  Clerk (Auth)    │         │  OpenAI / Pusher│
│  Prisma       │         │  Rekassa (Conf)  │         │  (внешние API)  │
└───────────────┘         └─────────────────┘         └─────────────────┘
```

**Связи:**
- Запросы к API идут через tRPC; контекст содержит `userId`, `organizationId` (из Clerk), `prisma`, `clerk`.
- Защищённые процедуры требуют аутентификации; `organizationId` используется для изоляции данных (мультитенантность).
- Документы и отчёты привязаны к организации; парсинг и сверка выполняются на сервере с использованием ИИ и утилит парсинга.

## Основные функциональные модули

| Модуль | Назначение | Краткое описание |
|--------|------------|------------------|
| **Auth** | Регистрация, вход, онбординг | Страницы sign-in/sign-up (Clerk), SSO callback, onboarding (профиль, организация). Компоненты: SignInForm, SignUpForm, OnboardingForm, LogoutButton. |
| **Reports** | Отчёты за период | CRUD отчётов, пошаговый мастер (форма отчёта: шаги import_info, import_docs, import_bank/crm, import_sales, import_expenses, import_result). Список отчётов с вкладками (в работе / готовые), создание и удаление. |
| **Documents** | Загрузка и разбор файлов | Загрузка Excel; парсинг (xlsx + ИИ: тип документа, банк, строка заголовков, колонки); создание Document и Transaction; слияние операций КНП (merge-knp-transactions). |
| **Reconciliation** | Сверка банк ↔ CRM | Сопоставление банковских и CRM-транзакций по отчёту; создание/обновление Reconciliation; типы операций (TransactionType); ИИ для предложений сверки; модальные окна создания/редактирования сверок. |
| **Transactions** | Транзакции и типы операций | Таблица транзакций отчёта; справочник типов операций (доход/расход); привязка типа к сверке. |
| **Connection (Rekassa)** | Интеграция с кассой | Сохранение и получение зашифрованных учётных данных Rekassa (rekassaRouter: saveCredentials, getCredentials); UI подключения в кабинете. |
| **Server AI** | ИИ-обработка документов | detectBank, detectTableStartRow, detectTableColumns, detectImage, reconcileDocs; используются в documents и reconciliation. |
| **Webhook / API routes** | Внешние интеграции | Приём вебхуков (receiveTransaction); create-payment (Pusher); payment-result, proxy при необходимости. |
| **Organization / User** | Организации и пользователи | Управление организацией и привязкой пользователей (UserOrganization); профиль пользователя в кабинете. |
